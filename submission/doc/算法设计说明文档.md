# 智能代理算法设计说明文档

## 1. 任务理解

本赛题要求实现一个移动端 GUI Agent。Agent 在每一步接收用户指令、当前安卓手机截图、历史动作等信息，并输出下一步标准动作。合法动作包括 `CLICK`、`TYPE`、`SCROLL`、`OPEN` 和 `COMPLETE`。其中，点击和滑动坐标均需要使用归一化到 `[0, 1000]` 范围内的相对坐标。

移动端 GUI Agent 的主要难点包括：

1. 移动端界面元素密集，搜索框、按钮、底部导航、弹窗、输入法状态容易混淆；
2. 多步任务依赖历史动作，单张截图往往不足以判断当前进度；
3. 多模态模型的原始输出格式不稳定，可能包含自然语言解释、非标准 JSON、伪函数调用等内容；
4. 评测器严格检查 `action` 与 `parameters`，格式不合规会直接导致失败；
5. 提交包有大小限制，不适合引入本地 OCR、检测模型或复杂视觉模块。

因此，本方案采用轻量级的“模型决策 + 协议约束 + 动作后处理”框架：多模态大模型负责理解界面语义和任务目标，Agent 负责将模型输出转化为稳定、合法、可执行的标准动作。

## 2. 总体框架

本方案实现的 Agent 主要包含以下流程：

```text
AgentInput
  ├── 用户指令
  ├── 当前截图
  ├── step_count
  └── 历史动作
        ↓
首步 App 识别与 OPEN 快速路径
        ↓
构造多模态 Prompt
        ↓
调用官方 BaseAgent._call_api()
        ↓
解析模型原始输出
        ↓
动作标准化与坐标归一化
        ↓
任务状态感知的后处理修正
        ↓
返回标准 AgentOutput
```

提交代码主要由以下文件构成：

```text
src/
├── agent_base.py
├── agent.py
├── requirements.txt
└── utils/
    ├── __init__.py
    ├── action_parser.py
    ├── image_utils.py
    └── prompt.py
```

其中，`agent.py` 是核心 Agent 实现；`utils/prompt.py` 负责构造提示词和识别常见 App 名称；`utils/action_parser.py` 负责解析和标准化模型输出；`utils/image_utils.py` 保持轻量实现，用于图片输入处理。

## 3. 核心设计

### 3.1 首步 OPEN 快速路径

对于一键式移动端任务，第一步通常是打开目标 App。如果用户指令中明确出现目标 App，例如“百度地图”“美团”“爱奇艺”“去哪儿旅行”等，Agent 会在第一步直接输出 `OPEN` 动作。

该设计有两个优点：

1. 减少一次多模态模型调用，降低推理成本；
2. 避免模型在桌面截图上误点图标，提高首步稳定性。

该规则只在 `step_count == 1` 且历史为空时启用，避免任务执行过程中重复打开 App。

### 3.2 多模态 Prompt 约束

Agent 会将用户指令、当前截图和最近历史动作组织成多模态 Prompt。Prompt 中明确约束模型：

1. 只进行下一步动作决策；
2. 输出必须可解析为标准动作；
3. 坐标必须是 `[0, 1000]` 范围内的归一化整数；
4. 点击应选择控件中心位置；
5. 根据历史动作判断当前任务进度，避免重复点击或重复输入；
6. 对搜索、输入、确认、播放、下单等常见移动端流程进行单步判断。

通过这种方式，模型主要负责界面理解和目标定位，而 Agent 负责动作协议的稳定落地。

### 3.3 多格式动作解析

模型输出格式可能并不总是严格 JSON，因此本方案实现了鲁棒动作解析器。解析器支持多种输出形式，包括：

```json
{"action": "CLICK", "parameters": {"point": [500, 300]}}
```

```text
CLICK:[[500, 300]]
TYPE:['搜索内容']
SCROLL:[[500,800],[500,300]]
OPEN:['应用名称']
COMPLETE:[]
```

也支持包含自然语言解释的宽松输出。解析器会提取其中最可能的动作，并统一转换为标准格式。

动作别名也会被归一化，例如：

```text
tap / press / 点击    -> CLICK
input / 输入          -> TYPE
swipe / scroll / 滑动 -> SCROLL
open / 打开           -> OPEN
done / finish / 完成  -> COMPLETE
```

### 3.4 参数标准化与坐标处理

解析后的动作会进入 `sanitize_action()` 进行参数标准化：

```text
CLICK    -> {"point": [x, y]}
SCROLL   -> {"start_point": [x1, y1], "end_point": [x2, y2]}
TYPE     -> {"text": "..."}
OPEN     -> {"app_name": "..."}
COMPLETE -> {}
```

对于坐标，本方案进行以下处理：

1. 将小数坐标、百分比坐标和归一化坐标统一转换到 `[0, 1000]`；
2. 将越界值裁剪到合法范围；
3. 对明显异常点进行安全修正；
4. 对 App 名称进行常见别名归一化，例如“去哪旅行”归一化为“去哪儿旅行”。

这样可以减少由于格式差异、坐标尺度差异或别名差异导致的无效动作。

### 3.5 历史动作记忆

多步 GUI 任务中，当前截图相同或相似时，是否已经点击过搜索框、是否已经输入过关键词、是否已经进入详情页，对下一步动作有重要影响。

因此，Agent 会维护一个紧凑历史记忆，记录最近若干步动作及参数。该历史用于：

1. 避免重复打开 App；
2. 避免重复点击搜索图标；
3. 判断输入框是否已经聚焦；
4. 判断是否已经输入目标文本；
5. 判断任务是否已经接近完成。

历史信息也会被写入 Prompt，帮助模型理解当前执行阶段。

### 3.6 动作后处理与执行安全

多模态模型在移动端界面上容易出现一些高频错误，例如：

1. 重复点击搜索图标，而不是点击搜索输入框；
2. 输入完成后没有点击搜索确认；
3. 搜索结果页点击偏移；
4. 任务完成后继续点击；
5. `OPEN` 的 App 名称存在轻微别名差异。

本方案在模型输出之后加入轻量后处理逻辑，根据 App 类型、用户指令、step_count 和历史动作进行修正。该后处理不改变官方接口，只在输出前修正明显不稳定或不合规的动作。

这类规则主要服务于移动端 GUI 的常见操作模式，例如：

```text
打开 App -> 点击搜索入口 -> 聚焦输入框 -> 输入关键词 -> 点击搜索/确认 -> 选择结果 -> 完成
```

对于外卖、视频、地图、旅游、音频等场景，后处理规则可以显著降低重复点击、错误提交和过早完成的概率。

### 3.7 API 调用兼容性

所有模型调用均通过官方 `BaseAgent._call_api()` 完成。Agent 不直接硬编码 API Key，不在代码中保存密钥，也不绕过官方基类创建新的评测接口。调试阶段使用 `VLM_API_KEY`、`DEBUG_API_URL`、`DEBUG_MODEL_ID`；正式评测阶段由主办方环境提供评测相关配置。

## 4. 方案特点

本方案的主要特点如下：

1. **协议安全**：无论模型原始输出格式如何，最终都尽量转化为合法 `AgentOutput`；
2. **轻量实现**：不依赖本地 OCR、检测模型或额外视觉模型，提交包体积小；
3. **历史感知**：通过最近动作记忆判断任务进度，减少重复操作；
4. **坐标鲁棒**：支持多种坐标尺度输入，并统一裁剪到 `[0, 1000]`；
5. **App 别名归一化**：降低应用名称不一致导致的 `OPEN` 失败；
6. **移动端流程优化**：针对搜索、输入、确认、选择、完成等常见链路进行稳定化处理；
7. **接口兼容**：严格遵守主办方 `BaseAgent`、`AgentInput`、`AgentOutput` 协议。

## 5. 依赖说明

本方案使用 Python 3.10.12。提交包中的 `src/requirements.txt` 只声明 Agent 运行所需的直接依赖，而不是完整本地开发环境的 `pip freeze`。

最终依赖如下：

```text
openai==2.24.0
Pillow==12.1.1
numpy==2.2.6
pandas==2.3.3
openpyxl==3.1.5
```

其中，`openai` 用于调用 OpenAI-compatible 多模态接口；`Pillow` 和 `numpy` 用于基础图像处理；`pandas` 和 `openpyxl` 保留为本地评测及结果处理相关基础依赖。

## 6. 本地运行方式

本地调试时，可在 `src/` 目录下安装依赖并运行评测脚本：

```bash
cd src
pip install -r requirements.txt
export VLM_API_KEY="your_api_key"
export DEBUG_API_URL="your_debug_api_url"
export DEBUG_MODEL_ID="your_debug_model_id"
python test_runner.py --data_dir ./test_data/offline --output_dir ./output --no_debug_test
```

正式提交时不需要提交测试数据、输出目录或调试日志。主办方评测时会根据提交包中的 Agent 代码和依赖文件运行评测。

## 8. 总结

本方案以多模态大模型为核心决策器，以规则化协议约束、鲁棒解析、历史动作记忆和轻量后处理作为稳定执行层。该设计能够在不引入复杂本地视觉模型的前提下，提高移动端 GUI Agent 在多步任务中的动作合法性、流程稳定性和任务完成率，适合作为移动端智能代理的轻量、高效、可维护方案。
